# AI 编程指导经验

## 一、项目背景

本项目 Jiuj 是一个邮件智能助手应用，通过 AI 分析邮件内容，提取待办事项并展示在看板上。尽管有详细的 PRD（产品需求文档）和架构设计文档，但在实际开发过程中仍然遇到了大量问题。本文档旨在总结这些问题及其原因，形成指导经验，避免今后重蹈覆辙。

## 二、主要问题回顾

### 2.0 最新问题记录（2026-04-18）

**问题 1：邮件拉取重复，拉取策略问题**
- **现象**：每次都拉取最近 3 天的全部邮件，而不是只拉取新增邮件
- **根本原因**：
  - 只使用了 SINCE 日期过滤，没有利用 IMAP UID 机制
  - last_uid 虽然记录了，但没有用在搜索条件中
- **解决方案**：
  - 新增条件优先使用 UID 搜索：`UID {last_uid+1}:*` 只拉取新增邮件
  - 只有在首次拉取（last_uid=0）时才使用 SINCE 最近 3 天
- **经验教训**：
  - IMAP UID 是递增的唯一标识符，非常适合增量拉取
  - 优先使用 UID 机制，而不是只依赖日期

**问题 2：看板显示过期列表问题**
- **现象**：用户要求删除看板中删除过期列表，专注于"提醒"核心功能
- **根本原因**：功能范围理解偏差
- **解决方案**：
  - 删除 `item_status 枚举中删除 Overdue 状态
  - 看板只显示 pending 状态的事项
  - 按日期从近到远排序

**问题 3：看板分成两个可折叠列表**
- **现象**：用户要求把有 deadline 和只有 time 的事项分成两个列表
- **解决方案**：
  - 使用 `ChevronDown`/`ChevronRight` 图标控制折叠
  - 默认展开，点击标题切换
  - 两个列表独立折叠状态

**问题 4：提醒逻辑优化**
- **现象**：精确时间 vs 日期-only 事项的提醒策略
- **根本原因**：
  - 日期-only 事项（如 "2026-04-20" 不需要两段提醒
  - 精确时间（如 "2026-04-20T10:00:00" 适合两段提醒
- **解决方案**：
  - 新增 `is_date_only()` 判断函数检查时间是否为 00:00:00
  - 日期-only：在前一天 10:00 提醒一次
  - 精确时间：按用户设置的默认提醒偏移量
- **经验教训**：
  - 根据数据的粒度不同，提醒策略要灵活调整
  - 不要一刀切，要根据实际情况处理

**问题 5：添加 source_date 字段**
- **现象**：用户要求看板显示发件日期
- **解决方案**：
  - 模型中新增 `source_date` 字段
  - 数据库 schema 新增 `source_date` 列
  - 前端显示在发件人后
- **经验教训**：
  - 新增字段要同步更新所有层：模型 → 数据库 → Repository → 前端

**问题 6：完成按钮确认框不消失**
- **现象**：点击完成确认框的 "完成"按钮，确认框不关闭
- **根本原因**：
  -  `completeMutation` 的 `onSuccess` 回调中缺少 `setCompleteConfirm({ isOpen: false, itemId: null })`
  - `ignoreMutation` 有，但 `completeMutation` 没有
- **解决方案**：
  - 在 `completeMutation` 的 `onSuccess` 中添加关闭逻辑
- **经验教训**：
  - 类似功能要检查是否有遗漏的地方

**问题 7：跳过名单先检查后拉取**
- **现象**：先拉取完整邮件再检查跳过名单，浪费资源
- **根本原因**：
  - 顺序错误：先 FETCH RFC822（完整邮件），再检查跳过名单
- **解决方案**：
  - 改为：先 FETCH BODY.PEEK[HEADER.FIELDS (FROM)] → 检查跳过名单 → 再拉完整邮件
- **经验教训**：
  - 优化流程顺序，先做轻量检查，再做重操作

**问题 8：删除 attachments 字段**
- **现象**：attachments 字段已经没用了，但还在代码和数据库中
- **解决方案**：
  - 模型中删除 attachments 字段
  - 数据库 migration 删除 attachments 列
- **经验教训**：
  - 删除无用代码和字段要彻底

**问题 9：删除 unused constants**
- **现象**：有两个常量 `MAX_BATCH_SIZE` 和 `MAX_COMPLETED_ITEMS` 没用了
- **解决方案**：删除它们
- **经验教训**：
  - 定期清理 unused code

### 2.1 邮件拉取相关问题

**问题 1：邮件拉取卡住**
- **现象**：一直显示"正在连接邮箱"，没有其他反应
- **根本原因**：
  - IMAP 日期格式错误（使用中文月份，而 IMAP 协议要求英文月份缩写）
  - UID 搜索逻辑错误，导致拉取了 6000+ 封历史邮件，而不是最近 3 天
- **经验教训**：
  - 必须了解协议的具体要求（如 IMAP 的日期格式：DD-Mon-YYYY）
  - 不要硬编码日期，应该通过系统时间动态计算
  - 添加详细的日志输出，便于定位问题

**问题 2：拉取邮件数量过多**
- **现象**：第一次启动拉取了所有历史邮件
- **根本原因**：
  - 没有正确使用 IMAP 的 SINCE 命令
  - 双重保障不足
- **解决方案**：
  - 使用 SINCE 命令 + 本地日期过滤双重保障
  - 明确限制只拉取最近 3 天的邮件

### 2.2 AI 分析相关问题

**问题 1："error decoding response body" 错误**
- **现象**：AI 分析时报错，无法解析响应
- **根本原因**：
  - AI API 响应格式与预期不符
  - 中文字符截取导致字节边界错误（使用字节计数而非字符计数）
- **经验教训**：
  - 先读取响应文本并记录日志，再进行反序列化
  - 处理中文字符时必须使用 chars() 而非字节操作

**问题 2：AI 分析过程中 panic**
- **现象**：分析到第 3/3 批次后一直转圈
- **根本原因**：中文字符截取导致 panic
- **解决方案**：使用 chars().count() 和 take() 进行字符截取

**问题 3：截止日期显示为 "{date}"**
- **现象**：看板显示 "截止：{date}" 占位符
- **根本原因**：
  - AI 返回了 "{date}" 占位符
  - 没有对这种情况进行处理
- **解决方案**：
  - 在 ai_analyzer.rs 中检测并将 "{date}" 设为 None
  - 前端只在 deadline 存在时显示

**问题 4：日期信息未融入内容**
- **现象**：截止日期和参会日期单独显示一行，用户希望融入内容
- **根本原因**：AI 提示词不够明确
- **解决方案**：
  - 修改 AI 提示词，明确要求将日期信息包含在 content 字段中
  - 提供具体示例说明期望的输出格式

### 2.3 看板显示相关问题

**问题 1：终端显示有信息但看板显示为 0**
- **现象**：数据库中已有数据，但前端查询不到
- **根本原因**：
  - status 字段存储为 JSON 字符串（如 "\"pending\""），而非简单字符串（如 "pending"）
  - 查询条件不匹配
- **经验教训**：
  - 数据格式必须一致
  - 添加详细日志追踪查询过程

**问题 2：邮箱地址后显示 UUID**
- **现象**：看板中邮箱地址后显示多余的 UUID 字符
- **根本原因**：存储了 account_id（UUID）而非邮箱地址
- **解决方案**：修改 app_scheduler.rs，存储邮箱地址

**问题 3：邮件信息与发件人不匹配**
- **现象**：批量分析邮件后，item 关联的邮件信息错误
- **根本原因**：批量分析后通过内容匹配关联 item 与邮件，导致匹配错误
- **解决方案**：改为逐个分析邮件，确保每个 item 正确关联到对应的邮件

**问题 4：已完成数量无法显示**
- **现象**：前端看板未正确显示已完成事项数量
- **根本原因**：completedItems 查询的 enabled 条件设置不当
- **解决方案**：移除 enabled 条件，使其始终查询

### 2.4 功能删除相关问题

**问题：暂停/恢复功能删除不彻底**
- **现象**：用户要求删除暂停/恢复功能，但删除不彻底
- **根本原因**：没有全面搜索和删除所有相关代码
- **经验教训**：
  - 删除功能前，先全面搜索相关代码（使用 Grep 工具）
  - 确保删除前端、后端、API 等所有层的相关代码

### 2.5 AI 设置相关问题

**问题 1：AI 设置页面按钮无反应**
- **现象**："测试连接"和"保存"按钮点击后无反应
- **根本原因**：
  - 类型错误（tauri-api.ts 中类型不匹配）
  - 测试代码硬编码在项目中
- **经验教训**：
  - 不要将测试代码硬编码到项目中
  - 确保类型定义正确

**问题 2："No active AI profile found" 警告**
- **现象**：AI 分析时未找到激活的 AI 配置文件
- **根本原因**：
  - 新添加的 AI profile 没有自动设为 active
  - 初始化时没有检查并激活现有 profile
- **解决方案**：
  - 插入第一个 AI profile 时自动设为 active
  - 在 main.rs 中添加初始化逻辑

### 2.6 UI 交互相关问题

**问题："已完成"按钮位置调整**
- **现象**：用户希望保留功能，但改变显示位置
- **根本原因**：没有准确理解用户需求
- **经验教训**：
  - 仔细理解用户的每个要求
  - 区分"功能删除"和"位置调整"

## 三、问题产生的根本原因分析

### 3.1 对技术细节了解不足

1. **协议细节**：不了解 IMAP 协议的具体要求（如日期格式）
2. **字符编码**：对中文字符的处理方式不熟悉（字节 vs 字符）
3. **框架特性**：对 Tauri、React Query 等框架的特性了解不够深入

### 3.2 需求理解不准确

1. **过度解读**：有时过度解读用户需求，导致删除了不该删除的功能
2. **理解偏差**：对"删除"、"调整位置"等需求理解有偏差
3. **细节忽略**：忽略了用户提到的一些细节要求

### 3.3 代码质量问题

1. **测试代码残留**：测试代码硬编码在项目中
2. **数据格式不一致**：status 字段存储格式前后不一致
3. **错误处理不完善**：对 AI 返回的异常情况（如 "{date}"）没有处理

### 3.4 开发流程问题

1. **缺少充分测试**：修改后没有充分测试就交付
2. **缺少代码审查**：没有对修改进行全面审查
3. **急于修改**：在没有完全定位问题的情况下就急于修改

### 3.5 AI 辅助编程的局限性

1. **缺乏全局视野**：AI 可能只关注局部修改，忽略对其他部分的影响
2. **测试不充分**：AI 生成的代码可能缺少充分的测试
3. **对特定领域知识不足**：对 IMAP 协议等特定领域知识了解不足

## 四、指导经验

### 4.1 开发前准备

#### 4.1.1 充分理解技术栈
- 在开始开发前，务必了解所使用技术栈的关键特性和限制
- 对于涉及协议（如 IMAP）的部分，要仔细阅读协议文档
- 了解框架的最佳实践和常见陷阱

#### 4.1.2 仔细阅读 PRD 和架构设计
- 不要只看一遍，要反复阅读
- 标记出关键功能点和技术要求
- 确认自己理解了所有需求

#### 4.1.3 制定开发计划
- 将大任务分解成小任务
- 每个任务都要有明确的验收标准
- 预估每个任务的时间

### 4.2 开发过程中

#### 4.2.1 坚持"非必要，不修改"原则
- 只修改与当前任务相关的代码
- 不要"顺手"修改其他代码
- 如果确实需要修改其他部分，先征得用户同意

#### 4.2.2 添加详细日志
- 在关键节点添加日志输出
- 日志要包含足够的信息，便于定位问题
- 使用适当的日志级别（INFO、DEBUG、WARN、ERROR）

#### 4.2.3 充分测试
- 每个功能完成后都要测试
- 不仅测试正常情况，还要测试异常情况
- 测试边界条件

#### 4.2.4 先定位问题，再修改
- 遇到问题时，先充分分析和定位问题
- 使用日志、调试工具等辅助定位
- 不要急于修改，确保找到根本原因

#### 4.2.5 小心处理数据格式
- 确保数据格式前后一致
- 对输入数据进行验证
- 对边界情况进行处理

### 4.3 与 AI 协作的最佳实践

#### 4.3.1 明确指令
- 给 AI 的指令要具体、明确
- 提供上下文信息
- 说明期望的输出格式

#### 4.3.2 分步骤任务
- 将大任务分解成小步骤
- 每完成一步就检查结果
- 确认正确后再进行下一步

#### 4.3.3 审查 AI 生成的代码
- 不要盲目信任 AI 生成的代码
- 仔细审查代码逻辑
- 检查是否有安全隐患
- 确认代码符合项目规范

#### 4.3.4 测试验证
- AI 生成的代码必须经过测试
- 不仅测试功能，还要测试性能和稳定性
- 检查是否有副作用

### 4.4 处理 AI 相关功能

#### 4.4.1 提示词工程
- 提示词要清晰、具体
- 提供示例说明期望的输出
- 明确说明哪些信息是必需的
- 要求 AI 将相关信息整合到合适的字段中

#### 4.4.2 处理 AI 输出的不确定性
- 不要假设 AI 的输出总是正确的
- 对 AI 的输出进行验证
- 处理异常情况（如占位符、格式错误等）
- 有降级方案

#### 4.4.3 错误处理
- 捕获并处理 AI API 调用的错误
- 记录详细的错误日志
- 给用户友好的错误提示
- 有重试机制（如果适用）

### 4.5 删除功能的注意事项

#### 4.5.1 全面搜索
- 在删除功能前，先全面搜索相关代码
- 使用 Grep 工具搜索关键词
- 检查前端、后端、API 等所有层

#### 4.5.2 确认影响范围
- 确认删除该功能不会影响其他功能
- 检查是否有其他地方依赖该功能
- 考虑数据迁移问题（如果适用）

#### 4.5.3 逐步删除
- 可以先注释掉相关代码，测试确认无误后再删除
- 删除后进行全面测试
- 确认没有遗漏

### 4.6 UI/UX 相关

#### 4.6.1 准确理解用户需求
- 仔细聆听用户的每个要求
- 有疑问时及时确认
- 区分"功能删除"和"位置调整"
- 理解用户的真实意图

#### 4.6.2 保持界面一致性
- 遵循现有的设计风格
- 使用统一的组件和样式
- 保持交互方式的一致性

#### 4.6.3 用户反馈
- 及时向用户展示修改结果
- 收集用户反馈
- 根据反馈调整

## 五、总结

尽管有详细的 PRD 和架构设计，在使用 AI 辅助编程时仍然会遇到很多问题。这些问题主要源于：

1. **技术细节了解不足**：对协议、框架、字符编码等技术细节了解不够
2. **需求理解不准确**：对用户需求的理解有偏差
3. **代码质量问题**：测试代码残留、数据格式不一致等
4. **开发流程问题**：缺少充分测试、急于修改等
5. **AI 辅助编程的局限性**：缺乏全局视野、测试不充分等

通过遵循上述指导经验，我们可以：
- 减少问题的发生
- 提高开发效率
- 提升代码质量
- 更好地与 AI 协作

记住：AI 是一个强大的工具，但最终的质量把控还需要人来完成。
