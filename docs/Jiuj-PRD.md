# Jiuj — 产品需求文档（PRD）

> **Jiuj：啾啾——有邮件来了，该做的事别忘了。**

---

## 1. 产品定位

### 一句话

一个轻量级 Tauri 桌面工具，常驻后台，自动读取你的邮箱，用 AI 提取重要信息，该提醒的时候 Jiuj 你一下。

### 产品名释义

**Jiuj**（/dʒuːdʒ/）——来自小鸟啾啾的叫声（啾啾）。像一只安静蹲在系统托盘里的小鸟，时不时啾一声提醒你："嘿，有封邮件里有重要的事。"

### 核心价值

邮件里藏着任务、日程、截止日期、需要回复的事项——但你每天收到几十封邮件，总会不小心漏掉。

Jiuj 替你盯着邮箱。它不是邮件客户端，不展示邮件列表，不帮你回邮件。它只做一件事：**从邮件里捞出对你重要的东西，在合适的时间提醒你。**

### 产品特征

| 特征 | 说明 |
|------|------|
| 轻 | 四个字母，一个工具，一件事，安装包不到 20MB |
| 安静 | 后台运行，不烦你，有事才啾一声 |
| 安全 | 只读邮箱，不发不改，数据全在本地，Rust 后端内存安全 |
| 灵活 | 通过 Skill 自定义 AI 的分析偏好和提醒规则 |
| 开源 | MIT 协议，代码透明，自己可控 |

---

## 2. 目标用户

- 日常工作任务和信息主要通过邮件获取的人
- 邮件量大，容易遗漏重要事项或截止日期
- 希望有一个安静的后台小工具帮自己盯着邮箱
- 对数据安全有要求，不信任让 AI 访问自己的电脑
- 希望自定义"什么算重要"、"什么时候提醒"——而不是被产品决定

## 3. 核心流程

```
用户配置邮箱（IMAP）、AI 模型（API Key）、Skills（分析偏好）
            ↓
    Jiuj 后台定时拉取新邮件（默认每 **15 分钟**一次，用户可在 5/15/30/60 分钟中选择）
            ↓
    ┌─ 步骤 1：本地预处理（不调用 LLM）──────────┐
    │                                             │
    │  对所有新邮件：                               │
    │    是否在跳过名单？ → 是 → 跳过此邮件         │
    │           ↓ 否                              │
    │    本地脱敏处理（手机号/身份证/银行卡等）       │
    │           ↓                                 │
    │    格式化为固定 JSON 结构                      │
    │    {from, subject, date, body, accountId}     │
    │                                             │
    └─────────────────────────────────────────────┘
            ↓
    ┌─ 步骤 2：分批调用 LLM（每批最多 10 封）──────┐
    │                                             │
    │  汇总已启用的 Skill，动态拼接 Prompt：         │
    │  系统模板 + Skill 原文 + 本批邮件 JSON 数组    │
    │           ↓                                 │
    │  一次性调用 LLM，批量分析整批邮件              │
    │  LLM 返回 JSON 数组 [{事项1}, {事项2}, ...]   │
    │           ↓                                 │
    │  新事项存入本地 SQLite                        │
    │                                             │
    │  （本批失败时跳过该批，记录日志，继续下一批）    │
    └─────────────────────────────────────────────┘
            ↓
    本轮所有批次处理完毕，汇总本轮新增事项
            ↓
    合并为一条桌面通知（"本次拉取发现了 M 个新事项"）
    事项正常存储
            ↓
    ┌──────────────────────────────────────────┐
    │  看板展示                                   │
    │  按时间/优先级排列，用户标记完成 / 忽略       │
    │                                            │
    │  到期提醒引擎（每分钟扫描）：                 │
    │  事项截止前 → 按两段式提醒时间发送桌面通知     │
    │  （默认：提前 1 天 + 提前 2 小时）            │
    └──────────────────────────────────────────┘
```

**批量处理说明：**

| 场景 | 处理方式 |
|------|---------|
| 一次拉取到 N 封新邮件 | 先本地预处理（跳过名单 + 脱敏），再按每批 10 封分批调用 LLM，批量分析 |
| LLM 调用失败 | 按错误类型分类处理：①**网络类错误**（超时、连接失败）：跳过该批次，所有成功批次处理完后重试失败的批次（最多重试 1 次）；②**认证/配额类错误**（API Key 无效、余额耗尽 401/429）：立即停止本轮分析，不再重试，通过托盘图标标记错误状态；③**解析类错误**（LLM 返回非JSON格式）：该批邮件记为空结果并记录日志，不重试 |
| N 封邮件提取到 M 个新事项 | M 个事项全部存入数据库后，合并为一条桌面通知（"本次拉取发现了 M 个新事项"） |
| 邮件积压超过 10 封 | 自动分批（每批 10 封），依次处理。避免单次 Prompt 过长 |
| 单封邮件连续导致批次失败 | 自动隔离该邮件（移出当前批次），记录日志，其余邮件继续处理 |
| 认证/配额类错误 | 立即停止本轮，托盘图标变红 |
| 部分批次解析失败 | 正常存储成功的批次；失败的邮件不生成事项，用户可在日志中查看 |

## 4. 功能说明

### 4.1 邮箱管理

| 功能 | 说明 |
|------|------|
| 添加邮箱 | 输入邮箱地址 + IMAP 服务器 + 授权码 |
| 多邮箱支持 | 可添加多个邮箱账号，统一管理 |
| 拉取频率 | 可配置（默认 15 分钟，可选 5/15/30/60 分钟） |
| 仅读取 | 只通过 IMAP 读取邮件，永远不会发送、标记、修改邮件 |

### 4.2 AI 信息提取

| 功能 | 说明 |
|------|------|
| 批量分析 | 每 10 封邮件合并为一次 LLM 调用，大幅减少 API 调用次数和 token 消耗 |
| 提取内容 | 默认提取任务、日程、截止日期、需要回复的事项、重要通知。用户可通过 Skill 自定义提取规则 |
| 可见性过滤 | LLM 返回每条事项的 `visible` 字段，系统仅存储和展示 `visible: true` 的事项，过滤掉无需关注的邮件 |
| 活动时间 | LLM 区分 `deadline`（截止日期）和 `time`（活动/会议时间），两者独立提取 |
| 来源追溯 | 每个提取事项可追溯到来源邮件 |
| 提醒偏移 | LLM 可按 Skill 定义返回每条事项的 `remind_offsets`（提醒提前量，分钟数组），未返回时使用用户设置的默认值 |

### 4.3 看板

| 功能 | 说明 |
|------|------|
| 事项列表 | 展示所有提取出的事项，按时间和紧急程度排列。每个事项卡片显示：内容、优先级色标、截止时间/活动时间、**来源邮箱账号** |
| 事项详情 | 点击可查看来源邮件摘要、提取的关键信息、命中的 Skill |
| 标记完成 | 用户标记后移入已完成列表 |
| 忽略 | 不感兴趣的事项可删除/忽略 |
| 已完成折叠 | 已完成的事项折叠展示，保持界面清爽 |
| 事项上限 | 已完成 + 已忽略事项超过 **100条**时，自动清理超过 30 天的旧记录，并在看板底部显示提示"已自动清理 N 条过期已完成项" |

### 4.4 提醒通知

#### 4.4.1 新事项通知

| 功能 | 说明 |
|------|------|
| 新事项通知 | 提取到新事项时，桌面弹窗通知（多条合并为一条） |
| 系统原生通知 | 使用 macOS / Windows 原生桌面通知样式（Tauri notification 插件） |



#### 4.4.2 到期提醒（智能区分）

提醒逻辑根据事项时间精度区分：

| 类型 | 说明 | 提醒方式 |
|------|------|---------|
| **精确时间事项** | `deadline` 或 `time` 包含具体时间（小时/分钟） | 两段式提醒：提前 1 天 + 提前 2 小时（可配置） |
| **日期-only事项** | 只有日期，没有具体时间（如"2026-04-20"） | 单次提醒：截止日前一天上午 10:00 |

**优先级：** Skill 定义的值 > 用户界面设置的默认值 > 系统兜底值

**边界情况处理：**

| 场景 | 处理方式 |
|------|---------|
| 两段提醒时间都已过期 | 合并为一次立即通知（不连续弹两次） |
| 仅第一段已过期，第二段未到 | 第一段立即通知，第二段按时通知 |
| 无截止时间的事项 | 不触发到期提醒，仅在新事项提取时通知一次 |
| `remind_offsets` 为空数组 `[]` | 该事项不触发到期提醒 |

### 4.5 Skills 系统

Skills 是 Jiuj 的核心差异化特性。用户通过编写纯文本 **SKILL.md** 文件来定义 AI 的行为——**提取什么、怎么判断优先级、什么情况提醒、什么情况跳过。**
不提供任何的字符串匹配和规则匹配方法来解析SKILL.md，因为 skills 允许用户创建，如果用户的格式有误，就会导致系统出错。更为关键的是，大模型完全有能力理解完整的SKILL.md 文档，所有硬编码的解析规则都是多余的。

#### 4.5.0 内置默认 Skill

Jiuj 内置一个 `default` Skill，提供通用分析能力，用户无需配置即可开箱即用。内置 Skill 不能被删除，但用户可通过编辑器查看其内容。用户创建并启用新 Skill 后，`default` 自动变为非激活状态。

> **注意：** v1.0 仅包含一个内置默认 Skill。更多预设技能（重要邮件识别、工作提醒、新闻简报过滤、购物通知等）计划在 v1.1 提供，详见[第 10 节](#10-后续迭代方向v11)。

#### 4.5.1 Skill 文件格式

每个 Skill 是一个目录，包含一个 `SKILL.md` 文件：

```
~/.jiuj/skills/
├── professor/
│   └── SKILL.md          # 大学教授的分析偏好
├── deadline-hawk/
│   └── SKILL.md          # 专注于截止日期的激进提醒
└── weekly-digest/
    └── SKILL.md          # 每周汇总而非即时提醒
```

#### 4.5.2 SKILL.md 结构

```markdown
# Skill: professor

## identity
大学教师，主要关注教学安排、科研事务、行政通知。

## extract-rules
优先提取以下类型的信息：
- 截止日期（任何需要提交的时间）
- 会议安排（时间 + 地点 + 议题）
- 学生事务（答辩、选课、考试相关）
- 行政通知（职称评审、项目申报截止）
- 忽略：纯转发邮件、节日问候、会议纪要

## priority-rules
- 含"截止""deadline""due"且距到期 ≤ 48h → high
- 含"请回复""RSVP"且来自非 noreply 地址 → medium
- 含"通知""公告"但无明确行动项 → low
- 满足以下条件之一的事项提升一个优先级：
  - 发件人为院长、系主任、教务处
  - 主题含"紧急""urgent"

## notify-rules
- high 优先级：立即通知
- medium 优先级：正常通知
- low 优先级：不主动通知，仅在看板展示
- 截止日期前 24h、2h 各提醒一次

## custom-prompt
除了默认的提取指令外，额外关注：
- 邮件中是否要求填写表格/表单
- 是否涉及财务报销或经费审批
- 是否涉及设备/实验室预约
```

#### 4.5.3 用户Skill 

- 不支持**同时启用多个 Skill**
- 在 skill 看板上选择的时候，是单选，不能多选
- 选择之后，系统提示词拼接时，采用新选择的 skill
- 新Skill只对新拉取的邮件立即生效；已分析的邮件不会自动重新分析，用户可手动触发"重新扫描最近邮件"来重新分析
- 禁用当前激活的 Skill 时，自动切换为默认 Skill，确保始终有 Skill 在工作

#### 4.5.4 Skill 管理 UI

| 功能 | 说明 |
|------|------|
| 浏览 Skill | 列出所有已安装的 Skill，显示名称、描述、启用状态 |
| 启用/禁用 | **单选按钮（Radio）**控制当前启用的 Skill。同一时间只能有一个 Skill 处于激活状态，选择新 Skill 时自动取消之前的激活状态。内置 default Skill 作为兜底：禁用当前激活的 Skill 时自动切回 default |
| 编辑 | 在应用内打开 SKILL.md 编辑器（内置轻量 Markdown 编辑器） |
| 新建 | 从模板创建新 Skill |
| 导入/导出 | 支持导入SKILL.md，或导出当前 SKILL.md。v1.0 **不做安全扫描**（见 §9），后续版本再引入基础安全扫描（检测 URL 注入、指令覆盖等），届时对外部来源的 Skill 显示 **"来自外部"** 警告标识 |
| 排序 | 拖拽调整 Skill 的显示顺序 |

### 4.6 数据脱敏

发送给 LLM 前，本地正则自动替换敏感信息：

| 脱敏内容 | 示例 | 脱敏后 |
|----------|------|--------|
| 手机号 | 13812345678 | [手机号] |
| 身份证号 | 310...1234 | [身份证号] |
| 银行卡号 | 6222...5678 | [银行卡号] |
| 邮箱地址（正文中） | zhangsan@xxx.com | [邮箱地址] |
| IP 地址 | 192.168.1.1 | [IP地址] |

#### 4.6.1 邮件正文截断

为避免超长邮件（如含长邮件链）撑爆 LLM 的 context window，在发送给 LLM 前对邮件 body 进行截断：

| 规则 | 说明 |
|------|------|
| 单封上限 | 每封邮件 body 最多保留 **8000 字符**（UTF-8 字符数，非字节数） |
| 截取策略 | 超过上限时，**直接从尾部截断**，保留前 8000 字符 |
| 标记 | 被截断的邮件在 JSON 中附带 `"truncated": true` 标记 |
| HTML 处理 | 若邮件为 HTML 格式，先去除 HTML 标签转为纯文本后再计算字符数和截断 |
| Prompt 提示 | 系统 Prompt 中包含说明："以下邮件内容可能被截断，以 `[已截断]` 标记表示" |

### 4.7 跳过名单

| 功能 | 说明 |
|------|------|
| 跳过发件人 | 指定某些发件人地址，其邮件直接跳过，不提取不上传 |
| 跳过域名 | 指定某些邮件域名（如 @noreply.xxx.com），批量跳过 |
| 快捷设置 | 在主页看板中，来源邮件之后，添加一个按钮“加入跳过名单”，点击按钮之后，需要弹出确认弹窗|

### 4.8 AI 设置

| 功能 | 说明 |
|------|------|
| 多配置管理 | 支持创建多个 AI 配置（Profile），每个配置独立保存厂商、模型、API Key。同一时间只有一个配置处于激活状态 |
| 厂商选择 | 内置厂商：OpenAI / DeepSeek / Kimi（月之暗面）/ 智谱（GLM）/ 通义千问（Qwen）/ Claude / Ollama（本地模型） |
| 自定义厂商 | 用户可手动添加任意兼容 OpenAI API 格式的厂商，填写名称、API Base URL、默认模型名即可 |
| API Key | 每个 AI 配置独立存储 API Key，本地 AES-256-GCM 加密存储，支持粘贴操作。**首次保存时**弹出友好提示："即将安全加密存储您的凭证" |
| 模型选择 | 为每个厂商提供描述性模型标签（如"GPT-4o"、"DeepSeek Chat"等） |
| 测试连接 | 一键测试 API 是否可用 |
| 切换激活 | 在多个 AI 配置之间切换当前激活的配置，切换后立即生效 |



### 4.9 系统托盘

| 功能 | 说明 |
|------|------|
| 最小化到托盘 | 关闭主窗口后，Jiuj 缩到系统托盘，后台继续运行 |
| 托盘菜单 | 右键菜单：打开看板 / 暂停拉取 / 退出 |
| 状态指示 | 托盘图标显示当前状态（正常/暂停/错误） |
| 退出 | 从托盘退出 = 完全退出应用，停止所有后台任务 |

## 5. 界面设计

### 5.1 国际化（i18n）

Jiuj 从第一版起内置中/英双语支持：

| 设计原则 | 说明 |
|----------|------|
| 语言跟随系统 | 默认跟随操作系统语言，用户可手动切换 |
| 所有 UI 文本外置 | 所有界面文案存储在独立的 locale 文件中，不硬编码在代码里 |
| 可扩展 | 架构上预留接口，未来支持社区贡献更多语言包 |
| Skill 文案不受限 | SKILL.md 中的 `custom-prompt` 使用用户自己写的语言，不做翻译 |

**locale 文件结构：**

```
src/i18n/locales/
├── zh-CN.json          # 简体中文
├── en-US.json          # 英文
└── index.ts            # 语言加载逻辑
```

### 5.10 交互状态说明

所有页面均需处理以下交互状态：

| 状态 | 说明 | 处理方式 |
|------|------|---------|
| **正常态（Normal）** | 页面数据加载完成，可正常操作 | 展示完整 UI |
| **加载态（Loading）** | 数据请求中（首次打开页面 / 刷新 / 提交操作后） | 骨架屏（Skeleton），卡片区域显示灰色占位条 |
| **空状态（Empty）** | 无数据 | 参见 §5.9 各页面空态设计，含引导按钮 |
| **错误态（Error）** | 请求失败 / 连接异常 | 红色提示条 + 重试按钮 |
| **禁用态（Disabled）** | 按钮不可点击 | 按钮置灰 + cursor: not-allowed。如：未保存时"测试连接"按钮不可点、内置 Skill 的删除按钮不可用 |
| **确认态（Confirm）** | 危险操作前二次确认 | 弹出 ConfirmDialog（删除账号 / 加入跳过名单 / 重置 Skill 等） |

> **全局错误处理**：所有 Tauri invoke 调用统一通过 `useTauriCommand` hook 捕获错误，以 Toast/Snackbar 形式展示给用户。网络类错误自动提供重试选项。

## 6. 技术架构

### 6.1 技术选型

| 组件 | 选型 | 说明 |
|------|------|------|
| 桌面框架 | Tauri v2 | Rust 后端 + Web 前端，安装包 < 20MB，内存占用低，无原生模块 ABI 问题 |
| 邮件拉取 | imap crate + mail-parser crate | Rust 原生 IMAP 客户端和邮件解析，静态编译无 ABI 兼容问题 |
| AI 分析 | 多厂商 API | 用户自配 Key，支持 OpenAI / DeepSeek / 智谱 / 通义 / Claude / Ollama |
| 数据存储 | SQLite（rusqlite） | Rust 原生 SQLite 绑定，静态编译进二进制，无 Node.js 原生模块问题 |
| 数据脱敏 | Rust regex | 纯 Rust 正则替换，不依赖外部服务 |
| 桌面通知 | tauri-plugin-notification | macOS / Windows 原生通知样式 |
| 系统托盘 | Tauri Tray API + tauri-plugin-notification | 后台常驻，托盘菜单 |
| 密码存储 | AES-256-GCM 加密 | 凭证加密后存入 SQLite 数据库，密钥持久化到 `~/.jiuj/.key` |
| 国际化 | i18next | 成熟的 i18n 方案，支持命名空间、插值、复数 |
| Skill 编辑器 | markdown-it + 自定义 parser | 仅用于编辑器 UI 的段落解析展示，LLM 分析管线不解析 SKILL.md |
| HTTP 客户端 | reqwest | Rust 生态主流 HTTP 库，用于 LLM API 调用 |

### 6.2 架构设计原则

| 原则 | 体现 |
|------|------|
| 简洁 | 核心流程清晰线性（拉取 → 脱敏 → 分析 → 存储 → 通知），没有多余的抽象层 |
| 可配置 | 通过 Skill 系统让用户自定义 AI 行为，核心代码不需要为每个个性化需求做改动 |
| 单一职责 | 每个模块只做一件事：mail-fetcher 拉邮件、sanitizer 脱敏、ai-analyzer 调 LLM、reminder-engine 管提醒 |
| 数据本地化 | 所有数据和配置都在用户本地，不依赖外部服务（LLM API 除外） |
| 轻量 | Rust 后端静态编译，无运行时依赖，安装包 < 20MB，内存占用 < 50MB |

### 6.4 数据库设计（SQLite）

**items 表（提取的事项）**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT (UUID) | 主键 |
| content | TEXT | 事项内容 |
| deadline | TEXT (ISO 8601) | 截止时间（可为空） |
| time | TEXT (ISO 8601) | 活动/会议时间（可为空）。与 deadline 区分：deadline 是"在此时间前完成"，time 是"在此时间参加" |
| priority | TEXT | 紧急程度（high/medium/low） |
| item_type | TEXT | 事项类型（task/deadline/reply/notification/other） |
| remind_offsets | TEXT | JSON 数组，提醒提前量（分钟），如 `"[1440, 120]"`。为空时使用用户设置的默认值 |

> **优先级硬性约束（LLM Prompt 中必须体现）：** 有截止时间（deadline 不为空）的事项，**优先级不可为 low**。理由：有明确时间节点的事项天然具备紧迫性，low 优先级仅适用于无截止时间的纯信息通知类事项。

> **notify-rules 与两段式到期提醒的关系：**
> Skill 中的 `notify-rules` 定义的是**新事项提取时的即时通知策略**（high→立即弹、medium→正常弹、low→不弹），而 **§4.4.2 的两段式到期提醒是独立运行的定时引擎**，两者互不冲突：
> - 两段式提醒引擎**只看 deadline 和 remind_offsets**，不看 priority，也不受 notify-rules 影响
> - 所有有 deadline 且 remind_offsets 非空数组 `[]` 的事项，都会被两段式引擎扫描和提醒
> - notify-rules 只控制"刚提取出来那一刻要不要弹窗告诉用户"
| notified_stages | TEXT | JSON 数组，已通知的阶段索引，如 `"[0, 1]"` 表示两段都已提醒 |
| source_email_id | TEXT | 来源邮件 ID |
| source_from | TEXT | 发件人 |
| source_subject | TEXT | 邮件主题 |
| source_account | TEXT | 来自哪个邮箱账号 |
| source_date | TEXT | 发件日期（ISO 8601 格式） |
| matched_skill | TEXT | 命中的 Skill 名称（可为空，单选，同时只有一个生效） |
| status | TEXT | 状态（**pending** / **completed** / **ignored**）。共 3 种状态：
  - **pending**（进行中）：默认状态，事项正常待处理
  - **completed**（完成）：用户手动标记为已完成
  - **ignored**（忽略）：用户手动标记为忽略 |

### items 状态流转图

```
                    ┌─────────────┐
     用户操作        │   pending    │ ← 新建事项的初始状态
      (标记完成)      │  （进行中）    │
           ┌────────→│   🔵 蓝色    │
           │         └──────┬──────┘
           │                │ 截止时间到达 + 未完成/未忽略
           │                ▼
           │         ┌─────────────┐
           │         │   overdue    │
           │         │   （过期）    │
           │         │   🔴 红色    │
           │         └──────┬──────┘
           │                │
           │    ┌───────────┼───────────┐
           │    ▼                       ▼
           │ ┌────────┐            ┌────────┐
           │ │completed│            │ ignored │
           └─│ （完成）│            │ （忽略） │
             │ 🟢 绿色 │            │ ⚪ 灰色 │
             └────────┘            └────────┘

  pending → completed：用户点击"完成"
  pending → ignored：   用户点击"忽略"
  overdue → completed：用户在过期项上点击"完成"（还原）
  overdue → ignored：   用户在过期项上点击"忽略"
```

### items 状态颜色与图标规范

| 状态 | 颜色 | 前景色 | 图标建议 | 说明 |
|------|------|--------|---------|------|
| **pending** | 🔵 蓝 (#3B82F6) | 白色文字 | ⏰ 时钟 / 📋 文档 | 进行中，待处理 |
| **overdue** | 🔴 红 (#EF4444) | 白色文字 | ⚠️ 警告 | 已过期，需关注 |
| **completed** | 🟢 绿 (#22C55E) | 白色文字 | ✅ 对勾 | 已完成 |
| **ignored** | ⚪ 灰 (#9CA3AF) | 深灰文字 | ⊘ 忽略符号 | 已忽略 |

> **设计原则**：红色 = 需要立即关注（overdue），蓝色 = 正常进行中，绿色 = 已解决，灰色 = 不再关心。
> 过期项在看板中应**置顶显示**且带轻微脉冲动画或加粗边框，确保用户第一眼就能注意到。
| last_notified_at | TEXT (ISO 8601) | 上次通知时间（可为空） |
| created_at | TEXT (ISO 8601) | 创建时间 |
| completed_at | TEXT (ISO 8601) | 完成时间（可为空） |

**accounts 表（邮箱账号）**

> **注意：** 邮箱授权码和 API Key 均以 AES-256-GCM 加密形式存入 SQLite 数据库（见 §7 安全与隐私）。AccountRepo 不涉及明文密码的读写。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT (UUID) | 主键 |
| email | TEXT | 邮箱地址 |

## 7. 安全与隐私

| 措施 | 说明 |
|------|------|
| 只读邮箱 | 通过 IMAP 只读取邮件，永不发送、标记、修改 |
| 数据本地化 | 所有数据存储在用户本地电脑（SQLite），不上传任何服务器 |
| 密码加密 | 邮箱授权码和 LLM API Key 均使用 **AES-256-GCM 加密后存入 SQLite 数据库**。加密密钥（AES 密钥）由应用启动时生成，单独存储在本地配置文件（`~/.jiuj/.key`）中，仅受操作系统文件权限保护。**首次保存时**弹出友好提示："即将安全加密存储您的凭证"，避免用户被系统授权弹窗吓到 |
| 脱敏处理 | 发送 LLM 前自动脱敏手机号、身份证、银行卡、邮箱地址、IP 地址 |
| 跳过名单 | 用户可指定不处理的发件人/域名，这些邮件不会被提取和上传 |
| Skill 安全 | Skill 是纯文本指令文件，不包含可执行代码。Skill 中的 custom-prompt 仅作为 LLM prompt 的一部分，不直接执行 |
| Rust 内存安全 | 后端使用 Rust 语言，编译期保证内存安全和线程安全，无缓冲区溢出等风险 |
| 零执行权 | 不执行任何系统命令，不访问文件系统（除 Skill 文件读取和 SQLite 操作） |
| 不收集数据 | 不向任何第三方发送用户数据（LLM API 除外） |
| Tauri 安全模型 | 默认最小权限，前端无法直接访问系统 API，所有操作通过 Tauri command 白名单控制 |

## 附录 A：LLM Prompt 构建流程

Jiuj 的 AI Prompt 由系统模板 + 内置 Skill + 用户 Skill 动态拼接：

```
┌─────────────────────────────────────┐
│ 1. 默认系统 Prompt 模板                     │
│    "你是一个邮件信息提取助手..."       │
├─────────────────────────────────────┤
│ 2. Skill 原文                       │
│    （已启用 Skill 的完整 SKILL.md）    │
├─────────────────────────────────────┤
│ 3. 批量邮件内容（脱敏后，每批最多 10 封）  │
│    JSON 数组：[{from, subject, date,  │
│    body, accountId}, ...]            │
└─────────────────────────────────────┘
            ↓
    一次性发送给 LLM → 返回 JSON 事项数组
```

### 默认系统 Prompt 模板

```
你是一个邮件信息提取助手。请从以下邮件中提取所有重要信息。

提取规则：
- 提取任务、日程安排、截止日期、需要回复的事项、重要通知
- 忽略纯转发邮件、节日问候、自动通知（除非含明确行动项）

优先级规则：
- **有明确截止时间的事项 → high 或 medium（不可为 low）**：有deadline天然具备紧迫性
- 含明确截止时间且距到期 ≤ 48h → high
- 有截止时间但距到期 > 48h → medium
- 无截止时间 + 含"请回复""RSVP" → medium
- 无截止时间 + 仅为信息通知 → low

> **重要**：low 优先级仅适用于无截止时间的纯信息通知类事项。有 deadline 的事项最低为 medium。

对每条信息，请以 JSON 格式输出：
{
  "items": [
    {
      "content": "事项描述（不超过20字，含关键日期时间）",
      "deadline": "2026-04-15T10:00:00+08:00（如有截止日期）",
      "time": "2026-04-17T13:30:00+08:00（如有活动/会议时间）",
      "visible": true,
      "priority": "high/medium/low",
      "item_type": "task/deadline/reply/notification/other",
      "remind_offsets": [1440, 120]
    }
  ]
}

字段说明：
- content：事项描述，不超过20字，必须包含关键日期和时间信息
- deadline：截止日期，ISO 8601 格式（如 2026-04-15T10:00:00+08:00）。无截止时间时为 null
- time：活动/会议时间，ISO 8601 格式。与 deadline 区分：deadline 是"在此时间前完成"，time 是"在此时间参加"。无活动时间时为 null
- visible：该事项是否需要显示在看板上。有截止日期或活动时间或需要回复时为 true；普通通知、广告、无明确行动项时为 false
- priority：优先级。**有 deadline 的事项不可设为 low。low 仅用于无截止时间的信息通知**
- item_type：事项类型（task/deadline/reply/notification/other）
- remind_offsets：提醒提前量（分钟数组）。如 [1440, 120] 表示提前 1 天和提前 2 小时。
  如果该事项有特殊的提醒需求，可自定义此数组。
  如果无需特殊提醒，设为 null（将使用用户默认提醒设置）。
  如果该事项不需要提醒，设为 []（空数组）。

如果没有提取到任何重要信息，返回 {"items": []}。

{custom_prompt_sections}

> **custom_prompt_sections 替换规则**：
> - 若当前 Skill 的 `custom-prompt` 段落有内容 → 替换为该内容原文
> - 若当前 Skill 无 `custom-prompt` 段落或该段为空 → **移除整行 `{custom_prompt_sections}`**（不保留空行）

以下是需要分析的邮件（共 {email_count} 封，部分邮件内容可能被截断以控制长度）：
---
{emails_json}
---

请逐封分析上述邮件，提取所有重要信息，返回 JSON 数组。
```

## 附录 B：Skill 编写指南（面向用户）

### 最小可运行的 Skill

```markdown
# Skill: my-rules

## identity
我是一个开发者，关注项目截止日期和代码审查请求。

## custom-prompt
额外关注：GitHub PR review 请求、CI/CD 失败通知。
```

### 完整 Skill 参考

| 段落 | 必填 | 说明 |
|------|------|------|
| `identity` | 否 | 描述用户身份，帮助 AI 理解上下文 |
| `extract-rules` | 否 | 自定义提取规则（追加到默认规则之后） |
| `priority-rules` | 否 | 自定义优先级判断规则（追加到默认规则之后） |
| `notify-rules` | 否 | 自定义通知策略。LLM 会据此决定每条事项的 `remind_offsets`（提醒提前量），系统按此值执行提醒 |
| `custom-prompt` | 否 | 任意额外的 prompt 内容，直接追加到 LLM 指令中 |

> **注意：** 所有段落都是可选的。一个只包含 `custom-prompt` 的 Skill 就是合法的。未填写的段落使用内置默认值。
